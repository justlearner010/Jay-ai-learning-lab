# 开发任务说明模板

> 用途：在让自己或 AI 修改已有仓库前，先把任务边界、输入输出和验收标准说清楚，避免改偏、改大或改到无关文件。

## 1. 任务描述

- Feature:
- Goal:
- Expected behavior:
  - 输入:
  - 输出:

## 2. 设计约束

- 保持现有接口不变，除非任务明确要求修改。
- 如果需要新增逻辑，优先隔离在新的函数或模块中。
- 避免跨模块副作用。
- 优先使用简单实现，不做无关抽象。

## 3. 实现前检查

- 需要修改哪些函数或模块？
- 输入和输出契约是什么？
- 至少 3 个边界情况是什么？
- 可能的失败模式是什么？

## 4. 测试要求

- 如果已有测试：确保现有测试通过。
- 如果新逻辑缺少覆盖：只补充和本次功能相关的测试。
- 如果没有测试框架：不要新建完整测试体系，只做必要的最小 sanity check。

## 6. 输出要求

完成后说明：

- 改了什么
- 改了哪些文件
- 如何验证
- 是否有未完成或需要后续确认的事项

## 最小示例

```text
Feature: 添加每日学习记录模板

Goal: 让我每天可以用固定格式记录学习内容，方便之后复盘。

Expected behavior:
输入: 新建一篇 daily log
输出: 文件包含“今日目标、学习内容、遇到的问题、明日计划”四个部分。

Files in Scope:
templates/daily-log.md
README.md
```
